# 公开版与 GitHub 双重审核同步设计

日期：2026-07-15
版本：v1
项目：`primary-knowledge-system`
状态：等待用户书面确认

## 1. 目标

把当前本地小学知识系统制作成可公开协作的版本，并公开经过版权清理的数学、语文、英语三科基础知识。社区可以通过 GitHub 持续补充知识；合并后的公共变更由本地系统自动拉取，再经过本地模型和规则二次审核，只有通过的内容才进入本地正式知识库。

公开协作不得上传学生作业、错题、掌握情况、照片、姓名、学校或其他家庭数据。

## 2. 仓库结构

采用两个公开仓库和一个本地私有层：

1. `primary-knowledge-system`：公开系统代码、数据契约、LLM Wiki 规则、审核器、同步器、CLI、测试和看板接口。
2. `cn-primary-knowledge-base`：公开三科基础知识，按 `subjects/math`、`subjects/chinese`、`subjects/english` 分区；每个知识实体使用统一稳定 ID 和统一来源契约。
3. 本地私有层：学生数据、本地观察、未公开来源、待公开候选和个人配置，不进入任何公开仓库。

系统代码与知识数据分仓，便于独立发布版本、控制许可证边界，并避免下载系统代码时强制获取全部知识内容。

## 3. 双重审核

### 3.1 第一关：GitHub 公共审核

社区只能通过 Pull Request 修改公共知识库。Pull Request 必须通过：

- schema、稳定 ID、引用闭合和目录边界校验；
- 重复实体、别名冲突和关系环路检查；
- 学科、实体类型、规范年级和教材位置检查；
- 来源完整性、来源等级和原创表达检查；
- 隐私、教材正文、习题、答案、插图和连续摘录扫描；
- 维护者对知识真实性、来源和归类的最终审核。

只有已经合并到受保护 `main` 分支的提交，才被视为公共发布候选。未经审核的分支、Fork、Issue 和 Pull Request 内容不得进入本地同步入口。

### 3.2 第二关：本地模型与规则审核

公共提交同步到本地后不得直接写入正式库。每一个新增、修订、合并、关系变化和删除操作都必须同时经过：

- 本地确定性规则：schema、身份、查重、来源、年级、教材位置、关系、版权字段和隐私字段；
- 本地模型判断：语义等价、知识真实性、表达质量、学科归属、年级合理性、来源是否支持主张，以及是否与现有知识冲突。

规则或模型任一项未通过，内容都不能进入本地正式知识库。模型不可用、返回结构无效或超时，也视为未完成审核，不允许降级绕过。

### 3.3 审核结果

本地二审产生三种结果：

- `ACCEPTED_LOCAL`：写入事件账本并物化到正式 Obsidian、SQLite 和图谱。
- `QUARANTINED_LOCAL`：保留公共原始内容、commit、来源、模型结论和规则失败原因，但不进入正式查询与图谱。
- `PENDING_LOCAL_REVIEW`：模型暂时不可用或审核任务未完成，进入持久队列，不进入正式库。

隔离内容不自动删除。后续规则、来源或模型更新后可以重新审核；通过时生成新的接纳事件，不修改原始公共提交记录。

## 4. 同步流程

```text
社区 Pull Request
  -> GitHub CI 与维护者审核
  -> 合并到公共数据仓库 main
  -> 发布 commit 或 release
  -> 本地同步器发现新版本
  -> 下载到只读 incoming 快照
  -> 计算与上次已处理版本的实体差异
  -> 本地规则审核
  -> 本地模型审核
  -> 接纳 / 隔离 / 待审核
  -> 事件账本
  -> Obsidian / SQLite / 图谱物化
```

GitHub 不能主动写入离线电脑。本地同步器在电脑运行时定时轮询，也支持手动执行。同步器只读取公共仓库，不上传本地私有层。

每个公共 commit 只处理一次；每个实体变更使用 `(repository, commit, entityId, contentHash)` 作为幂等指纹。进程中断后从未完成实体继续，不重复提交已经接纳的事件。

## 5. 删除与回退

公共仓库中的删除、合并和降级同样属于知识变更，必须经过本地二审。公共删除不能直接删除本地正式实体：

- 审核通过后生成可恢复 tombstone；
- 审核未通过时进入隔离区，旧实体继续有效；
- 公共版本回退不会用旧快照覆盖本地事件历史；
- 本地可以从事件账本恢复任意已删除实体。

## 6. 公共数据边界

允许公开：

- 原创知识点名称、解释、掌握表现、常见误区和关系理由；
- 稳定 ID、类型、学科、年级范围、别名和结构化关系；
- 课程标准和教材的短定位、出版信息、版本、册次、单元和页码等事实元数据；
- 来源 URL、访问日期、证据等级和核验结论；
- 权利状态明确的公版古诗词原文及符合相应许可证的转录材料。

禁止公开：

- 现代教材正文、课文、例题、习题、答案、解析、插图、扫描件和教师用书内容；
- 足以连续还原教材或受保护工具书的摘录；
- 学生姓名、作业图片、手写内容、作答、错题、掌握状态和家庭观察；
- 无法说明来源和权利边界的第三方整库复制内容。

## 7. 许可证

- 系统代码采用 Apache License 2.0。
- 项目原创知识数据和文档采用 CC BY 4.0。
- 明确来源于 Wikisource 等 CC BY-SA 材料的转录内容单独标记并遵循 CC BY-SA 4.0。
- 课程标准、出版社页面和其他第三方材料保留其自身权利状态；来源登记不构成对第三方正文的重新许可。
- 每个实体携带 `licenseClass` 和 `sourceRefs`，公共导出器拒绝未声明权利类别的数据。

现有语文仓库已经具备代码与数据许可证。数学和英语仓库在公开前必须补齐许可证、NOTICE 和逐类权利边界；不能仅把仓库改为 public 就视为完成开源。

## 8. 本地与公共知识的关系

公共仓库是可共享的上游候选源，不是本地机器权威。本地事件账本仍是本地正式知识的机器权威。

- 公共内容通过本地二审后成为本地正式知识；
- 本地私有观察不会自动推送 GitHub；
- 本地产生、适合公开的新知识先进入 `public-candidate` 区；
- 公开候选经过隐私与版权导出检查后，生成 Pull Request，而不是直接推送到公共 `main`；
- Pull Request 合并后仍按普通公共变更再次经过本地二审，保证所有入口规则一致。

## 9. 反馈与隔离

隔离记录至少包含：

- 公共仓库、commit、Pull Request 和实体 ID；
- 原始 payload 与内容哈希；
- 命中的规则、模型版本、结构化判断和失败原因；
- 与本地现有实体的冲突或重复候选；
- 首次审核时间、重试次数和最终处理事件。

系统可以生成公开反馈报告，但默认不自动创建 GitHub Issue，也不得把本地私有内容放入反馈。人工确认后才允许发布反馈。

## 10. 发布与版本

两个公开仓库独立使用语义版本。数据仓库发布 manifest，包含：

- 数据版本、schema 版本和生成时间；
- 三科实体数、关系数和来源数；
- 每个文件的内容哈希；
- 最低兼容系统版本；
- 相对上一版本的新增、修改、删除和合并摘要。

本地同步器优先跟随正式 release；开发模式可以显式跟随 `main`。默认不读取任意分支。

## 11. 与现有开放本地写入设计的关系

本规格只增加“公共 GitHub 内容进入本地”的外部边界，不撤销现有本地 Agent 开放调用能力。

- 本地 Agent 的新增和修订继续经过统一 LLM Wiki 与学科策略。
- 公共 GitHub 内容额外要求公共审核和本地二审。
- 无论来自本地 Agent、公共同步还是直接 Markdown 扫描，最终都必须形成事件后才能成为正式知识。
- 本规格与旧设计冲突时，公共仓库同步以本规格为准。

## 12. 验收标准

1. 公开系统仓库不包含 Vault、学生数据、密钥或本机绝对路径。
2. 公开数据仓库包含数学、语文、英语三个独立目录和统一 manifest。
3. 公共 Pull Request 未合并时，本地同步器无法读取其内容。
4. 合并内容在本地必须同时执行规则和模型审核。
5. 任一审核失败的实体进入隔离区，不出现在正式查询和图谱。
6. 模型不可用时进入待审核队列，不降级直入正式库。
7. 同一公共版本重复同步保持幂等。
8. 同步中断不损坏上一版本的正式库，并可断点续跑。
9. 公共删除必须经过本地二审并可恢复。
10. 本地私有数据不会被公共候选导出器或反馈报告带出。
11. 三科公开导出通过许可证、来源、隐私和禁止内容扫描。
12. 公共数据通过本地二审后能稳定物化到 Obsidian、SQLite 和知识图谱。

## 13. 实施顺序

1. 清理并发布系统代码仓库。
2. 建立统一三科公共数据仓库和数据契约。
3. 为数学、英语补齐许可证与 NOTICE，并执行三科公开数据审计。
4. 从三个现有学科库生成版权安全的公共数据快照。
5. 实现 GitHub release/main 拉取、差异计算和幂等同步。
6. 实现本地规则审核、模型审核、隔离区和持久重试队列。
7. 接通事件账本、Obsidian、SQLite 和图谱物化。
8. 建立 GitHub CI、Pull Request 模板、贡献指南和分支保护说明。
9. 使用真实三科数据完成离线、断点、重复同步、删除和隐私验收。
